Why should you care?

Real AI training corpora can be tens of gigabytes, far more than your laptop's RAM.

If you load the whole file at once, you crash before training even starts. The patterns in this module are exactly what lets real tools stream billions of samples through a model while keeping memory nearly constant:

  • Comprehensions for clean, fast transforms
  • Generators with yield to hand back one batch at a time
  • Lazy evaluation so nothing is computed until it is needed
Key insight

These same ideas power tools like PyTorch DataLoader and Hugging Face datasets, letting them train on data far larger than memory.

Part 1 · List Comprehensions

Replace for loops with one expression

A list comprehension collapses the build-an-empty-list, loop, filter, append pattern into a single line. An if clause keeps only the items you want. It makes intent obvious and runs faster than the equivalent loop.

# Old way (verbose for loop)
cleaned_lines_old = []
with open(filename, 'r') as f:
    for line in f:
        line = line.strip().lower()
        if len(line) > 0:
            cleaned_lines_old.append(line)

# Pythonic way (list comprehension)
with open(filename, 'r') as f:
    cleaned_lines = [
        line.strip().lower()
        for line in f
        if len(line.strip()) > 0
    ]
Watch the scale

Comprehensions are great for small datasets, but the list version still loads every cleaned line into RAM at once. Fine for thousands of lines, dangerous for millions.

Part 2 · Dict Comprehensions

Build vocabularies in one line

A dict comprehension has the same shape as a list comprehension but produces key/value pairs. In AI work you constantly build word-to-frequency maps and word-to-ID vocabularies.

# Old way: count words with a for loop
vocab_old = {}
for word in all_words:
    if word in vocab_old:
        vocab_old[word] += 1
    else:
        vocab_old[word] = 1

# Pythonic way: dict comprehension
unique_words = set(all_words)
vocab = {word: all_words.count(word)
         for word in unique_words}
Readable is not always efficient

all_words.count(word) rescans the whole list for every unique word. For large corpora reach for collections.Counter, which counts in a single pass.

Part 3 · The Memory Problem

Why loading everything breaks

Reading a whole file with readlines or a list comprehension puts every line in memory at the same time. That works for a 3KB sample and fails for a 10GB corpus on an 8GB machine.

# Loading everything at once
with open(filename, 'r') as f:
    all_lines = f.readlines()

mem_estimate = sys.getsizeof(all_lines)
print(f"Estimated memory: {mem_estimate / (1024*1024):.2f} MB for list object")
# Problem: what if the file is 10GB and you have 8GB of RAM?
Key insight

File to RAM to Process only scales while the file fits in RAM. Once data outgrows memory you need File to Generator to Batch to Process, discarding each batch before the next.

Part 4 · Generator Functions

yield: a lazy, pausable stream

A generator function uses yield instead of return. Each yield hands back one value and pauses the function, resuming exactly where it left off on the next request. A batch generator accumulates a few items, yields the batch, resets, and repeats.

def line_generator(filename, batch_size=100):
    with open(filename, 'r') as f:
        batch = []
        for line in f:
            cleaned_line = line.strip().lower()
            if len(cleaned_line) > 0:
                batch.append(cleaned_line)
                if len(batch) >= batch_size:
                    yield batch
                    batch = []
        if batch:            # don't forget the last partial batch
            yield batch
Two easy mistakes

Always create a fresh batch = [] inside the function and remember to yield the final partial batch after the loop, or you silently drop the tail of your data.

Part 4 · Generator Functions

Pause, emit, resume

Each yield pauses the function and returns a full batch. The single batch box is filled, emitted, then cleared, so only one batch lives in memory at a time.

The payoff

No matter how many batches you produce, memory holds just one batch at a time. That is what makes streaming a multi-gigabyte file possible.

Part 5 · Lazy Evaluation

Eager lists vs lazy generators

A list comprehension is eager: it computes and stores every value immediately. Swap the square brackets for parentheses to get a generator expression, which is lazy and computes nothing until you ask with next() or a for loop.

# Eager: stores all 1,000,000 squares now
squares = [x**2 for x in range(1000000)]

# Lazy: computes nothing yet, one at a time
squares_gen = (x**2 for x in range(10))
print(next(squares_gen))  # 0
print(next(squares_gen))  # 1
print(next(squares_gen))  # 4
Laziness has a catch

A generator is exhausted after one pass. Calling list(gen) a second time returns []. Use a generator for a single streaming pass, and convert to a list only when you need to iterate more than once.

Part 6 · Building a Data Stream

Compose cleaning and batching

The lab combines everything into an infinite data loader. An outer while True loop lets you re-stream the corpus for multiple training epochs, cleaning and filtering each line, yielding fixed-size batches, and stopping cleanly after max_batches.

def infinite_data_loader(path, batch_size=8, max_batches=None):
    batch_count = 0
    while True:                      # loop for multi-epoch training
        batch = []
        with open(path) as f:
            for line in f:
                line = line.strip()
                if not line or '\t' not in line:
                    continue
                content = line.split('\t', 1)[1].lower()
                batch.append(content)
                if len(batch) == batch_size:
                    yield batch
                    batch = []
                    batch_count += 1
                    if max_batches and batch_count >= max_batches:
                        return
Key insight

The while True wrapper is the trick behind infinite epochs: when the file ends the loop reopens it, so the loader can feed a training run for as long as you want without ever holding the full dataset in memory.

Part 6 · Building a Data Stream

The epoch loop

When the file ends, control feeds back to the top for a new epoch. Batches are emitted continuously, and max_batches is the exit gate.

Constant memory

No matter how many epochs or batches you produce, only one batch is ever held at a time. Memory stays flat from the first batch to the millionth.

Part 7 · PyTorch DataLoader

You just built a mini DataLoader

PyTorch's DataLoader is a production-grade generator built on the exact pattern from this module: it yields batches lazily, never loads the full dataset, and adds shuffling, sampling, and multi-process loading on top.

# The generator pattern behind DataLoader
def dataloader(dataset, batch_size):
    indices = list(range(len(dataset)))
    for i in range(0, len(indices), batch_size):
        batch_indices = indices[i:i+batch_size]
        yield [dataset[idx] for idx in batch_indices]

# Usage (lazy batching), just like the real thing:
for batch in dataloader(dataset, batch_size=32):
    train(model, batch)
The big idea

Comprehensions, yield, and lazy evaluation are not just style. They are the foundation that lets real systems train on datasets far larger than RAM, powering Hugging Face datasets and S3 streaming pipelines too.

What you learned

  • List comprehensions make data-cleaning transforms shorter, faster, and more readable than for loops
  • Dict comprehensions build vocabularies and frequency maps in one line, but watch for hidden O(n) scans like list.count
  • Generator functions use yield to pause and resume, streaming one batch at a time instead of loading everything
  • Lazy evaluation computes values only when requested, but a generator is exhausted after a single pass
  • Streaming and batching let you process files larger than your available RAM
  • An infinite while True loader supports multi-epoch training with constant memory
  • PyTorch DataLoader is a production generator built on these same principles
The big idea

Choose comprehensions for small in-memory work and generators for large one-pass streaming. That single choice decides whether your data pipeline fits in memory.
