Why should you care?

Before you can train a model or fine-tune an LLM, you need to store and find data quickly.

Picture GPT-4, which knows over 100,000 tokens. For every word it reads, it must:

  • Look the word up in its vocabulary
  • Find the matching token ID
  • Fetch the embedding vector for that ID
Key insight

The same task can take microseconds or whole seconds depending only on which data structure you pick. That single choice can decide whether an AI system is usable or unusably slow.

Part 1 · Lists

Lists: Ordered & Indexed

A list is an ordered, mutable sequence. Under the hood it is a dynamic array that stores references to objects, one after another.

  • Access by index is instant: O(1)
  • Searching by value is slow: O(n)
  • Great for: ordered sequences and iteration
  • Weak at: frequent lookups and existence checks

Part 1 · Lists

Working with lists

vocabulary = ["hello", "world",
              "neural", "network"]

# Indexing is O(1)
vocabulary[0]     # "hello"
vocabulary[-1]    # "network"

# Slicing makes a NEW list
vocabulary[:2]    # ["hello", "world"]

# Mutating in place
vocabulary.append("transformer")

# Searching is O(n), slow!
"neural" in vocabulary  # True (slow scan)

A few habits to build:

  • Reach for a list when order matters or you plan to iterate over everything
  • Remember that x in my_list quietly scans the whole list
  • Slicing never changes the original, it returns a fresh copy

Part 2 · Dictionaries

Dictionaries: Key-Value Pairs

A dictionary stores key-value pairs using a hash table. Instead of scanning, it jumps straight to the answer.

  • Lookup by key: O(1) on average, no matter how big it grows
  • Insert and delete: O(1)
  • Keys must be immutable (strings, numbers, tuples) and unique
  • Critical for: tokenizers, embeddings, caches, indexes
© mui-group

Part 2 · Dictionaries

How the magic works

A hash function turns a key into a number. That number points directly to a storage slot, so there is no searching at all.

token_dict = {
    "hello": 1045,
    "neural": 3421,
}

token_dict["neural"]          # 3421 (O(1))
token_dict.get("nope", -1)   # -1, no crash
token_dict["new"] = 99          # add in O(1)
Use .get()

Reading a missing key with [ ] raises a KeyError. Use .get(key, default) to return a fallback instead of crashing.

Part 3 · Performance

Try it yourself

Do not take our word for it. Measure it with time.perf_counter():

import time

vocab_size = 100000
vocab_list = [f"token_{i}" for i in range(vocab_size)]
vocab_dict = {f"token_{i}": i for i in range(vocab_size)}

target = "token_99999"   # worst case: the very last item

start = time.perf_counter()
target in vocab_list           # O(n): scans ~100,000 items
list_time = time.perf_counter() - start

start = time.perf_counter()
target in vocab_dict           # O(1): one hash lookup
dict_time = time.perf_counter() - start

print(list_time / dict_time)   # often hundreds to thousands x
Reading the result

The speedup is small for tiny collections but explodes as the data grows. This is exactly the gap that decides whether a model runs in real time.

Part 4 · Mutability

Mutable vs Immutable

Some objects can be changed after they are created. Others cannot. Knowing which is which prevents a whole category of bugs.

Immutable

  • Strings, tuples, numbers
  • Cannot change in place
  • Any "change" makes a new object
  • New memory address
  • Safe to share freely

Mutable

  • Lists, dicts, sets, arrays, tensors
  • Can change in place
  • Same object, new contents
  • Same memory address
  • Risk of surprise side effects

Part 4 · Mutability

Proving it with id()

The id() function returns an object's memory address. Watch what changes and what stays the same:

# Mutable: the list changes in place
vocab = ["hello", "world"]
print(id(vocab))     # 140234568100
vocab.append("AI")
print(id(vocab))     # 140234568100 ← SAME object

# Immutable: the string is replaced, not edited
text = "hello"
print(id(text))      # 140234567890
text = text + " world"
print(id(text))      # 140234567999 ← DIFFERENT object

Part 4 · Mutability

The aliasing trap

Assigning a mutable object to a new name does not copy it. Both names point to the same object, so a change through one is visible through the other.

# Both names, one list
a = ["neural", "network"]
b = a
b.append("transformer")

print(a)
# ['neural','network','transformer']
# a changed too! ⚠️

# Fix: make a real copy
b = a.copy()   # or list(a) or a[:]
Why it matters for ML

A preprocessing function that appends a pad token to its input will corrupt the original data on every call. Always ask: am I mutating the original, or working on a copy?

Reading it safely

# Direct access
r["model"]
# "gpt-4"

r["usage"]["total_tokens"]
# 87

r["choices"][0]["message"]["content"]
# "Data structures are..."

# Safe access with a default

Reading tips:

  • A [0] in the path means you hit a list, so grab an item by position first
  • Chain .get() calls when a key might be missing to avoid a crash
  • json.dumps(obj, indent=2) pretty-prints the whole structure so you can see the shape

Part 6 · Big O

A Big O cheat sheet

Big O describes how work grows as the input n gets larger. Here is where common operations land, from best to worst:

ComplexityNameExampleVerdict
O(1)ConstantDict lookup, array indexIdeal for AI
O(log n)LogarithmicBinary searchExcellent
O(n)LinearList search, single loopOften fine
O(n log n)LinearithmicEfficient sortingGood
O(n²)QuadraticNested loopsAvoid at scale
O(2ⁿ)ExponentialBrute-force branchingImpractical
Why it matters for LLMs

GPT-4 processes up to a million tokens per request, each needing a vocabulary lookup. With O(1) dictionaries that is about 0.1 seconds; with an O(n) list it would take hundreds of seconds.

Part 7 · The Lab

Steps 1–3: text to counts

Lowercase the text, split it into words, then count how often each word appears using collections.Counter:

import re
from collections import Counter

def tokenize_text(text):
    text = text.lower()
    return re.findall(r"\b[a-z]+\b", text)   # words only, no punctuation

words = tokenize_text(corpus)      # ['neural', 'networks', 'are', ...]
word_freq = Counter(words)         # {'neural': 12, 'networks': 9, ...}

word_freq.most_common(3)          # [('neural', 12), ('networks', 9), ...]
Why Counter?

Counter is a specialized dictionary. Each freq[word] += 1 is an O(1) operation, so counting the whole corpus is O(n) overall.

Part 7 · The Lab

Step 4: assign token IDs

Reserve a few special tokens first, then give every word an integer ID. Building the reverse map lets us decode later.

# Special tokens live at the start of every real tokenizer
SPECIAL_TOKENS = {"[PAD]": 0, "[UNK]": 1, "[BOS]": 2, "[EOS]": 3}

word_to_id = SPECIAL_TOKENS.copy()
next_id = len(SPECIAL_TOKENS)

for word, freq in word_freq.most_common():   # frequent words get smaller IDs
    if word not in word_to_id:
        word_to_id[word] = next_id
        next_id += 1

id_to_word = {i: w for w, i in word_to_id.items()}   # reverse map
TokenMeaning
[PAD]Fills short sequences so a batch is rectangular
[UNK]Stands in for any word not in the vocabulary
[BOS] / [EOS]Mark the beginning and end of a sequence

Part 7 · The Lab

Step 5: encode and decode

Wrap the two dictionaries in a small class. encode turns text into IDs; decode turns IDs back into text.

class SimpleTokenizer:
    def __init__(self, word_to_id, id_to_word):
        self.word_to_id = word_to_id
        self.id_to_word = id_to_word
        self.unk_id = word_to_id["[UNK]"]

    def encode(self, text):                       # text  → IDs
        return [self.word_to_id.get(w, self.unk_id) for w in tokenize_text(text)]

    def decode(self, ids):                        # IDs → text
        return " ".join(self.id_to_word.get(i, "[UNK]") for i in ids)

tok = SimpleTokenizer(word_to_id, id_to_word)
tok.encode("Neural networks learn")     # [5, 6, 14]
tok.decode([5, 6, 14])                  # "neural networks learn"
Notice the fallback

Both methods use .get() with a default so an unknown word maps to [UNK] instead of crashing. That is the aliasing and safe-lookup lesson paying off.

What you learned

  • Lists keep order and are great for iteration, but searching them is O(n)
  • Dictionaries give O(1) lookup through hashing, the backbone of AI performance
  • Mutable objects change in place; immutable ones are replaced, and id() reveals which
  • Aliasing bugs come from sharing a mutable object by accident, so copy when in doubt
  • Nested dictionaries model real API responses; read them one layer at a time
  • Big O predicts how code scales, and O(1) vs O(n) can mean seconds vs hours
  • A tokenizer is just two dictionaries turning text into numbers and back
The big idea

The right data structure is not a detail. It is what makes an AI system fast enough to be useful.
