Why should you care?

Every computer vision model, from ResNet to a modern vision transformer, expects its input in one very specific form: a fixed-size grid of normalized numbers.

Real photos come in wildly different sizes, color orderings, and value ranges. Feed them in raw and the model either:

  • Crashes on a shape mismatch, or
  • Simply fails to learn from unstable inputs
Key insight

The preprocessing pipeline you build here is the unglamorous but essential bridge between messy real-world images and a model that can actually train.

Part 1 · Images as Arrays

An image is a 3D grid
of numbers

A color image is a 3D NumPy array of shape (Height, Width, Channels). Height and Width are rows and columns of pixels; Channels holds the color intensities (usually 3). Each value is an 8-bit unsigned integer from 0 to 255.

import numpy as np

# A tiny synthetic color image: 100 rows, 200 cols, 3 channels
color_img = np.zeros((100, 200, 3), dtype=np.uint8)

print(color_img.shape)   # (100, 200, 3) -> (Height, Width, Channels)
print(color_img.dtype)   # uint8  (values 0 to 255)

# A grayscale image is just 2D: no channel axis
grayscale_img = np.zeros((100, 200), dtype=np.uint8)
print(grayscale_img.shape)  # (100, 200)
Watch the axis order

The (H, W, C) ordering trips people up because we usually say width before height. NumPy indexes rows (height) first, so img.shape[0] is height, not width. Getting this backwards is a classic source of bugs.

Part 2 · Loading Images

cv2.imread turns a file
into an array

OpenCV's cv2.imread reads an image file and returns it as a NumPy array. Inspecting .shape, .dtype, min, max, and mean tells you everything about the pixels.

import cv2

img = cv2.imread(image_path)   # returns a NumPy array, or None if not found

if img is None:
    print("Failed to load image")
else:
    print(f"Shape: {img.shape}")   # e.g. (480, 640, 3)
    print(f"Dtype: {img.dtype}")   # uint8
    print(f"Min:   {img.min()}")   # 0
    print(f"Max:   {img.max()}")   # 255
    print(f"Mean:  {img.mean():.2f}")
The silent None

Because cv2.imread returns None instead of throwing on a bad path, forgetting the None check leads to a confusing crash several lines later. Check immediately after loading.

Part 3 · The BGR Quirk

OpenCV loads colors
backwards

For historical reasons tied to early video hardware, OpenCV stores channels in Blue, Green, Red order instead of the Red, Green, Blue order that matplotlib, PIL, and PyTorch expect. The shape does not change, only the meaning of the three channels.

# OpenCV gives you BGR; convert to RGB for everything else
img_rgb = cv2.cvtColor(img_bgr, cv2.COLOR_BGR2RGB)

print(img_rgb.shape)   # unchanged, only channel order swapped

# Same center pixel, two orderings (Blue and Red are swapped)
bgr = img_bgr[img_bgr.shape[0]//2, img_bgr.shape[1]//2]
rgb = img_rgb[img_rgb.shape[0]//2, img_rgb.shape[1]//2]
print(f"BGR: {list(bgr)}")
print(f"RGB: {list(rgb)}")
The reflex to build

If a displayed image looks like red and blue are swapped (a blue sky turns orange), you forgot the conversion. Make cv2.cvtColor(img, cv2.COLOR_BGR2RGB) the reflex you run immediately after every cv2.imread.

Part 4 · Resizing

Squeeze every image
to a fixed 224x224

Neural networks need a fixed input size, but real images come in all dimensions. cv2.resize scales any image to a target size, filling in new pixel values by interpolation. INTER_LINEAR (bilinear) is the fast, good-quality default.

target_size = (224, 224)   # note: (width, height) order!

img_resized = cv2.resize(
    img_rgb, target_size,
    interpolation=cv2.INTER_LINEAR)

print(img_resized.shape)   # (224, 224, 3)
Argument order flips

cv2.resize takes its target as (width, height), the opposite of the (H, W) shape ordering.

Why 224?

224x224 is not arbitrary: it is the historical ImageNet convention that VGG, ResNet, and MobileNet all expect, a deliberate trade-off between detail and manageable computation. Newer models sometimes use other sizes (299x299, 512x512).

Part 5 · Normalization

Rescale pixels from
0 to 255 into 0.0 to 1.0

Models train far better when inputs live in a small, consistent range. We cast the integer pixels to float32 and divide by 255.0, mapping 0 to 255 into 0.0 to 1.0. This keeps features on a common scale and helps gradient descent converge.

img_normalized = img_resized.astype(np.float32) / 255.0

print(img_normalized.dtype)         # float32
print(img_normalized.min(),
      img_normalized.max())         # ~0.0 to ~1.0
print(f"Mean: {img_normalized.mean():.3f}")

# Advanced: some models also do ImageNet-style standardization
# img = (img - mean) / std   # per-channel mean and std
Divide by a float, not an int

Dividing by 255.0 (a float) not 255 matters: integer division would floor everything to 0. Forgetting to normalize is one of the top reasons a model "trains" but never converges, because 0 to 255 inputs produce huge, unstable gradients.

Part 6 · Batching

Stack many images
into one 4D array

Models process images in batches for efficiency. After running the same four steps on each image, collect the results in a list and use np.stack to combine them along a new first axis. The result is a 4D array of shape (N, H, W, C).

processed_images = []
for filename in image_files:
    img = cv2.imread(os.path.join(images_dir, filename))
    img = cv2.cvtColor(img, cv2.COLOR_BGR2RGB)
    img = cv2.resize(img, (224, 224), interpolation=cv2.INTER_LINEAR)
    img = img.astype(np.float32) / 255.0
    processed_images.append(img)

# Stack into a 4D batch: (N, H, W, C)
batch = np.stack(processed_images, axis=0)
print(batch.shape)   # e.g. (8, 224, 224, 3)
Why resize comes first

np.stack only works when every array has an identical shape, which is exactly why resizing (Part 4) has to happen before stacking. The leading N axis is the batch dimension that GPUs love.

Part 7 · Memory & Pipeline

Know your memory budget

A batch's size in memory grows linearly with N. One 224x224x3 float32 image is about 0.57 MB, so you can estimate any batch by multiplying. GPU memory is limited (often 4 to 24 GB), so batch size is a real constraint you tune.

Why powers of 2

Memory is the hidden reason batch size is often a power of 2: too big and you get an out-of-memory crash, too small and the GPU sits idle. Start small and increase while watching memory.

Part 7 · Memory & Pipeline

Own the whole pipeline

Wrapping the four steps in a single preprocess_image function makes the pipeline reusable and easy to reason about, and lets you estimate memory for any batch size.

def preprocess_image(image_path, target_size=(224, 224)):
    img = cv2.imread(image_path)                       # 1. load (BGR)
    img = cv2.cvtColor(img, cv2.COLOR_BGR2RGB)         # 2. BGR -> RGB
    img = cv2.resize(img, target_size,
                     interpolation=cv2.INTER_LINEAR)   # 3. resize
    img = img.astype(np.float32) / 255.0               # 4. normalize
    return img

# Memory grows linearly with batch size (float32 = 4 bytes/value)
for n in [32, 64, 128, 256]:
    mb = (n * 224 * 224 * 3 * 4) / (1024 * 1024)
    print(f"{n:3d} images: {mb:7.2f} MB")
The canonical pipeline

Load, BGR to RGB, Resize, Normalize, then Stack. One function captures the first four steps and works across any dataset.

What you learned

  • An image is a 3D NumPy array of shape (H, W, C) with uint8 pixels from 0 to 255; a batch is a 4D array (N, H, W, C)
  • cv2.imread loads images but returns None on a bad path and gives channels in BGR order, so always check for None and convert with cv2.cvtColor(img, cv2.COLOR_BGR2RGB)
  • cv2.resize takes its target as (width, height) and standardizes every image to a fixed size like 224x224, the ImageNet convention most pretrained models expect
  • Normalize by casting to float32 and dividing by 255.0 (a float, not an int) so pixels land in 0.0 to 1.0
