Why should you care?

You collected 80 rows of LLM prompt and response logs to train a quality-rating model, but the file is a mess.

  • Exact duplicate runs, plus missing prompts and ratings
  • Model names written five different ways (GPT-4, gpt4, GPT4)
  • Temperatures above 1.0 and dates in two formats
Key insight

Feed this straight into a model and duplicates will leak test data into training, missing values will crash it, and inconsistent labels will fragment what the model sees. Cleaning is the unglamorous work that decides whether your model learns real patterns or noise.

Our roadmap

We take the messy file through one deliberate step at a time, then export the result.

  • 1. Inspect the mess (profile the data)
  • 2. Remove exact duplicates
  • 3. Handle missing values (drop vs fill)
  • 4. Fix formatting (strings and standardization)
  • 5. Boolean indexing to filter invalid data
  • 6. Fix data types (dates and numerics)
  • 7. Split into train and test, then export clean CSVs

Part 1 · Inspect the Mess

Profile before you clean

You cannot fix what you have not measured. Load the data and take inventory: how many rows, which columns hold missing values, and how many exact duplicate rows exist. These counts become your baseline.

import pandas as pd

df = pd.read_csv("data/dirty_dataset.csv")
print(f"Loaded {len(df)} rows")

# Missing values per column (only those with any)
missing_counts = df.isna().sum()
print(missing_counts[missing_counts > 0])
print(f"Cells missing: {df.isna().sum().sum()}")

# Count exact duplicate rows
print(f"Found {df.duplicated().sum()} duplicates")
Key insight

Save baseline stats (initial rows, missing cells, duplicates) up front. Tracking data loss after each step is how you catch a filter that silently deleted half your dataset.

Part 2 · Remove Duplicates

Drop exact duplicate rows

Duplicate rows are the most dangerous kind of dirty data in ML: the same record can land in both training and test sets, leaking answers and inflating accuracy. .duplicated() flags repeats and .drop_duplicates() removes them, keeping the first occurrence by default.

# Peek at the duplicates before removing them
duplicated_mask = df.duplicated(keep=False)
print(df[duplicated_mask].sort_values("prompt_id").head(6))

# Remove exact duplicates (keeps first occurrence)
df = df.drop_duplicates()
print(f"Remaining rows: {len(df)}")
Key insight

keep=False shows every copy including the first, which is great for inspection. drop_duplicates(subset=[...]) lets you dedupe on an ID column when whole-row matching is too strict.

Part 3 · Missing Values

Drop critical fields, fill the rest

Missing data has no single fix. The decision depends on the column's role. Drop rows for fields you cannot train without; fill non-critical fields with a sensible default.

# Drop rows missing critical fields
df = df.dropna(subset=[
    "prompt_text",
    "response_text",
    "rating",
])

# Fill non-critical field with 0
df["tokens_used"] = df["tokens_used"].fillna(0)

print(f"Remaining rows: {len(df)}")
Field roleAction
Critical (prompt, response, target)dropna(subset=...)
Non-critical (tokens_used)fillna(default)
Why it matters for ML

Never blanket-fill the target variable you are trying to predict. Imputing a rating would teach the model on made-up labels, which is worse than dropping the row.

Part 4 · Fix Formatting

Standardize text so equal things look equal

To a model, 'GPT-4', 'gpt4', and 'GPT4 ' are three different categories even though they mean the same thing. Strip whitespace and map variants to one canonical form.

model_mapping = {
    "GPT-4": "gpt-4", "GPT4": "gpt-4", "gpt4": "gpt-4",
    "GPT-3.5": "gpt-3.5",
    "Claude-3": "claude-3", "Claude3": "claude-3",
    "Claude-3-Opus": "claude-3-opus",
}

df["model"] = df["model"].str.strip()      # remove whitespace first
df["model"] = df["model"].replace(model_mapping)

# Strip stray whitespace from all text columns
for col in ["prompt_text", "response_text", "category"]:
    df[col] = df[col].str.strip()

print(df["model"].unique())
Common trap

Strip whitespace before mapping. A trailing space means 'gpt4 ' will not match your mapping key 'gpt4' and the fix silently fails.

Part 5 · Boolean Indexing

Filter invalid data with masks

Boolean indexing is the workhorse of filtering. A comparison produces a True/False mask, and df[mask] keeps only the True rows. Combine conditions with & (and) and | (or), wrapping each in parentheses.

# Inspect what is invalid first
invalid_temp = df[~df["temperature"].between(0.0, 1.0)]
print(invalid_temp[["prompt_id", "model", "temperature"]])

# Keep only rows that pass every rule
df = df[
    df["temperature"].between(0.0, 1.0) &
    (df["tokens_used"] >= 0) &
    df["rating"].between(0.0, 5.0)
]
print(f"Remaining rows: {len(df)}")
Key insight

Each condition must be parenthesized and joined with & or | (not the Python keywords and/or). Use ~ to negate a mask, which is handy for previewing exactly what you are about to drop.

Part 6 · Fix Data Types

Make columns the type they should be

CSV loads everything loosely, so dates arrive as plain strings and numbers can end up as objects. pd.to_datetime with format='mixed' parses the two date formats, and .astype() coerces numeric columns so later math and comparisons behave.

# Dirty data mixes "2024-01-15" and "01/16/2024"
df["date"] = pd.to_datetime(df["date"], format="mixed")

df["temperature"] = df["temperature"].astype(float)
df["tokens_used"] = df["tokens_used"].astype(int)
df["rating"] = df["rating"].astype(float)
df["prompt_id"] = df["prompt_id"].astype(int)

print(df.dtypes)
Order matters

Fix types after filtering, not before. astype(int) throws if a column still contains NaN, so clean out missing and invalid rows first.

Part 7 · Split & Export

Shuffle, split 80/20, and save

A model must be tested on data it has never seen. Shuffle with a fixed seed, slice at the 80% index, then write both files.

Why shuffle

Raw logs are often ordered by time or category. Shuffling before slicing keeps the same mix of records in both the train and test halves.

Part 7 · Split & Export

The split in code

Shuffling with a fixed random_state makes the split reproducible, then slicing at the 80% index gives a train and a test set. Finally .to_csv(..., index=False) writes clean, ready-to-use files.

# Shuffle reproducibly, then slice 80/20
df_shuffled = df.sample(frac=1, random_state=42).reset_index(drop=True)
split_idx = int(0.8 * len(df_shuffled))

train_df = df_shuffled[:split_idx]
test_df = df_shuffled[split_idx:]

train_df.to_csv("data/output/Train.csv", index=False)
test_df.to_csv("data/output/Test.csv", index=False)
print(f"Train: {len(train_df)} rows, Test: {len(test_df)} rows")
Two habits to keep

Always set random_state so your split is reproducible, and always pass index=False to to_csv so pandas does not write a phantom index column that reappears as junk when you reload.

What you learned

  • Profile first: count rows, missing cells, and duplicates so you can measure every step's impact
  • Duplicates are the sneakiest ML risk because they leak test data into training and inflate accuracy
  • Handle missing values by role: drop rows for critical fields and the target, fill sensible defaults for the rest
  • Standardize text (strip then map) so semantically equal values are not treated as different categories
